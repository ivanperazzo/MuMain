Resumen ejecutivo (corregido). MuMain no tiene fixed timestep ni hilo de simulación: toda la lógica corre dentro de RenderScene(), 1× por frame renderizado ✓. La compensación es un factor global FPS_ANIMATION_FACTOR = clamp(differenceMs/40ms, 0, 1) (REFERENCE_FPS = 25), aplicado en 3363 sitios / 56 archivos. Dos mecanismos distintos explican tus dos síntomas:

① Lento <25 FPS: el clamp a 1.0 impide compensar bajo 25 FPS → velocidad = FPS/25 · normal (a 10fps = 40%). Afecta TODO. → tu "muchos jugadores → lento".
② Speedhack >25 FPS: el movimiento lineal es exactamente invariante sobre 25fps. Acelera lo que rompe esa invariancia: física compuesta (proyectiles), enteros/densidad, cámara cinemática, decays mal hechos. → tu "subo FPS → speedhack" en efectos.

1.Causa raíz — confirmada

Cadena del loop (freeze-on-skip confirmado)

Toda la simulación vive dentro de RenderScene. Si no renderiza, WorldTime y el mundo se congelan; solo la red y los timers wall-clock siguen. CalcFPS() tiene un único sitio de llamada (SceneManager.cpp:1046 ✓).

Mecanismo ① — lento bajo 25 FPS (probado)

// ZzzAI.cpp:725-730
const double differenceMs = std::max<double>(WorldTime - last, 0.001);
FPS = 1000.0 / differenceMs;
FPS_ANIMATION_FACTOR = std::clamp(float(REFERENCE_FPS / FPS), 0.f, 1.f);  // = clamp(differenceMs/40, 0, 1)

Distancia/seg = paso · Σfactor = paso · (Σ differenceMs)/40 = paso · 1000/40 = 25·paso — invariante mientras differenceMs ≤ 40ms (FPS ≥ 25). Cuando FPS < 25, differenceMs > 40 pero el factor se clampa a 1.0 → se pierde la compensación → velocidad = (FPS/25)·normal. A 10 fps = 40%. Idéntico efecto con un cap de FPS explícito.

Mecanismo ② — speedhack sobre 25 FPS (probado)

El movimiento lineal NO acelera (invariante, arriba). Acelera lo que rompe la invariancia: doble integración (cat D), enteros/densidad (cat C), sin escalar (cat E), decay roto (cat B✗). Detalle en §2.

Dependencia del monitor (runtime por defecto). EnableVSync() + SetTargetFps(-1) ⟹ el loop corre a Hz del monitor; la rama freeze solo dispara con cap explícito o ventana inactiva. ⟹ Mismo binario, físicas de efectos distintas según el monitor: 60 Hz → factor 0.42; 144 Hz → factor 0.17.

4.Rediseño del game loop (fixed timestep + render interpolado)

FIXED_DT = 40ms mantiene 25 tps idénticos al feel actual → 0 cambio de balance.

constexpr double FIXED_DT_MS = 40.0;   // 25 sim-ticks/s == REFERENCE_FPS
constexpr int    MAX_STEPS   = 5;      // anti spiral-of-death
double accumulator = 0.0, prevMs = Timer::NowMs();

while (!Destroy)
{
    double nowMs = Timer::NowMs(), frameMs = nowMs - prevMs; prevMs = nowMs;
    if (frameMs > 250.0) frameMs = 250.0;     // pausa/breakpoint: no acumular deuda

    PumpNetwork();        // drenar cola -> buffer de estado (no aplicar aún)
    SampleInput();        // SDL_PollEvent: acumular intención

    accumulator += frameMs;
    int steps = 0;
    while (accumulator >= FIXED_DT_MS && steps < MAX_STEPS) {
        SimulationSnapshotBegin();   // estado previo para interpolar
        FixedUpdate(FIXED_DT_MS);      // movimiento, path, animación lógica, efectos
        accumulator -= FIXED_DT_MS; ++steps;
    }
    if (steps == MAX_STEPS) accumulator = 0.0;   // descartar deuda

    const float alpha = float(accumulator / FIXED_DT_MS);   // 0..1
    if (CheckRenderNextFrame()) RenderScene(alpha, frameMs);
    else                       WaitForNextActivity();
}

Dentro de FixedUpdate el movimiento abandona el factor global (Position += Velocity_perTick). El render interpola: renderPos = lerp(prevTickPos, curTickPos, alpha). Cámara y blend de animación usan frameMs (dt real), NO alpha. Durante la migración, FPS_ANIMATION_FACTOR ≡ 1.0 para el código aún no portado.

6.Riesgos técnicos

  • Animaciones que asumen 25 fps (comentario ZzzAI.cpp:728): usar FIXED_DT=40ms evita re-tunear.
  • Cat D (re-integración): los arcos de proyectiles cambiarán al pasar a dt fijo — re-tunear constantes para igualar el feel a 25fps; testear alcance/altura.
  • Cat C (rediseño): convertir conteos enteros k/factor en tiempos/ticks; cuidar overflow ya existente (GOBoid BYTE).
  • Frame counters absolutos (iFrame, seeds GOBoid): mapear a tick count o WorldTime.
  • Interpolación / rubber-band: buffer de 1–2 ticks de retraso, interpolar (no extrapolar) entre estados servidor.
  • Desync con servidor: OpenMU es autoridad; la interp es cosmética, snap suave si divergencia > umbral.
  • Spiral of death: MAX_STEPS + clamp de frameMs lo cortan.
